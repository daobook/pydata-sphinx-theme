主题指南和操作指南#

这些部分涵盖了与开发这个主题相关的常见操作和主题。

制作发布版本#

这个主题使用 GitHub 标签和版本自动将新版本推送到 PyPI。有关此过程的信息，请参阅 发布清单。

更新 JavaScript 依赖项及其版本#

这个包的 javascript 依赖定义在 package.json 中，并分成几个类别，如 dependencies 和 `devDependencies。

要更新或添加 JS 依赖，修改（或附加）在每个部分中列出的包的列表。下次你构建文档（使用 nox 或 stb）时，这些新的依赖项将被安装并与主题捆绑在一起。

使用 nox#

下面是一些使用 nox 的额外提示。

参见

关于 CLI 可以启用的额外功能，nox 命令行文档 提供了许多有用的提示。

重装依赖#

要重新执行安装命令，请使用以下模式：

$ nox -s docs -- reinstall

或者完全清除由 nox 产生的环境，重新开始：

$ rm -rf .nox/docs

在你的全局环境中使用 nox#

如果你想在你的 global 环境（你称之为 nox 的环境）中使用 nox，你可以这样做：

$ nox --force-venv-backend none

# alternatively:
$ nox --no-venv

使用 none 将重用您当前的全局环境。详情请参阅nox 文档。

使用 pre-commit#

下面是一些使用 pre-commit 的技巧：

跳过预提交检查#

执行如下命令：

$ git commit --no-verify

对所有文件执行预提交#

默认情况下，pre-commit 会对提交中修改过的文件进行检查。要在所有文件上运行它，使用下面的命令：

$ pre-commit run --all-files

# Alternatively
$ pre-commit run -a

Web assets (CSS/JS/Fonts)#

这个主题包括几个网页资产，以简化开发和设计。我们的资产编译配置在 webpack.config.js 中。

编译和绑定 assets#

当 assets 被编译时，静态版本被放置在主题的静态文件夹的不同位置：

src/pydata_sphinx_theme/theme/pydata_sphinx_theme/static

对于许多 assets，在主题的 HTML 模板中会生成 <hash>，并将其添加到引用的末尾。这确保了在主题升级后，当浏览者返回您的站点时，能够提供正确的 asset 版本。

要编译 assets 并将其与主题绑定，运行以下命令：

$ nox -s compile

样式 (SCSS) 和脚本 (JS)#

CSS/JS assets 有两个相关的地方：

  • src/pydata_sphinx_theme/assets/styles 有 SCSS assets 的源文件。这些将被编译成 CSS。

  • src/pydata_sphinx_theme/assets/scripts 有 JS assets 的源文件。这些将被编译成 JS，并导入几个 vendored 的库（如 Bootstrap）。

  • src/pydata_sphinx_theme/theme/pydata_sphinx_theme/static 已经编译了这些 assets 的版本（例如 CSS 文件）。没有在 .git 中跟踪此文件夹，但它与主题的发行捆绑在一起。

Vendored 脚本#

除了我们自己的 CSS 和 JS，我们还提供几个包。例如，Bootstrap, JQuery 和 Popper。这是在 webpack.config.js 文件中配置的，并导入到 assets 文件夹中相应的 SCSS 或 JS 文件中。

FontAwesome 图标#

三种风格的 FontAwesome 5 Free 图标字体用于 图标链接 和警告，是唯一的 vendored 字体。

  • 它在 package.json 中作为依赖被管理。

  • 直接复制到网站的静态编译，包括许可证

  • 部分预加载，以减少早期图标呈现的 flicker 和工件

  • 在 webpack.config.js 中配置

Jinja 宏#

我们的 Webpack 构建会在 static/webpack-macros.html 文件中生成一组 Jinja 宏。

这些宏被导入到 main layout.html 文件中，然后插入到页面的各个位置以链接静态 assets。

一些 assets “preloaded”，这意味着浏览器在实际需要这些资源之前就开始请求这些资源。特别是，我们的 JavaScript assets 是在 <head> 中预加载的，而脚本实际上是在 <body> 的末尾加载的。

可访问性检查#

可访问性检查工具可以找到许多辅助技术无法帮助用户理解的常见 HTML 模式。

除了 CI 中的 Lighthouse 之外，pa11y 堆栈也是作为开发环境的一部分安装的。

关键组件：

  • pa11y使用无头浏览器（headless browser）来分析 HTML 页面，该页面有一组基于发布标准的可配置规则

  • pa11y-ci 在多个页面上运行 pa11y

  • pa11y-reporter-html 生成一些漂亮的 HTML 报告，适合审查

备注

目前使用的是 默认 pa11y 规则集 WCAG2AA，它是 Web 内容可访问性指南 的子集。快速参考 可能提供更轻松的阅读。

CI/CD 中的错误以及如何处理#

我们在文件 docs/scripts/a11y-roadmap.txt 中有一个已知的可访问性问题列表。这包含了错误列表，我们打算在将来修复这些错误，并且这些错误不会导致测试失败。

当在我们的 CI/CD 中运行 pa11y 可访问性审计时，它会检查任何不在这个列表中的错误，如果它发现了这些错误，就会导致作业出错。

当您在 CI/CD 作业中看到错误时，请查看 运行可访问性审计 作业下的日志。您应该会看到如下输出：

  WCAG2AA.Principle1.Guideline1_3.1_3_1.H43,H63: 1

导致错误的问题在 “不在路线图上” 部分。任何不在路线图上的错误都是我们在 PR 中意外引入的错误。这些错误应该被识别并修复。

修复可访问性错误#

我们在可 访问性路线图 中保留 known accessibility issues 的列表。这些问题目前已经被工具集标记，但是还没有被修复。

要开始处理可访问性路线图项目之一，注释掉 docs/a11y-roadmap.txt 中的一行，并重新运行审计以建立 baseline。

然后，在 HTML 模板、CSS 或 python 代码中修复问题，并重新运行审计，直到修复为止。

在本地运行可访问性审计#

要在本地运行可访问性问题查找程序，请执行以下操作：

$ nox -s compile  # Compile the theme assets
$ nox -s docs  # Build the documentation
$ python docs/scripts/a11y.py  # Run a helper script for an accessibility audit

最后一条命令的输出信息包括：

  • 我们试图维护的可访问性规则的当前状态的简短总结

  • JSON 和 HTML 报告的本地路径，其中包含所有发现的问题

更新我们的厨房水槽文档#

厨房水槽参考 是为了尽可能多地演示 Sphinx 构建的语法和风格。它直接从 sphinx-themes.org 文档 中复制而来，这样就可以使用与其他社区相比标准化的参考文档。这些页面的源文件存储在 sphinx-themes.org 存储库 中。

如果你想用 sphinx-themes.org 文件中的任何更改来更新我们的本地文件，只需将这些更改复制/粘贴到我们的本地文件并提交即可。

在 sphinx-themes.org 中，列出了一些页面，以及它们的来源：

  • index.rst (source)

  • api.rst (source)

  • lists-and-tables.rst (source)

  • paragraph-markup.rst (source)

备注

为了演示额外的样式和语法，使用 Theme Elements reference。
